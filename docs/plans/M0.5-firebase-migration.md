# M0.5 — Firebase Migration: Implementation Plan

**Milestone:** M0.5 (insertado entre M0 cerrado y M1 en curso)
**Status:** 🟡 In progress
**Goal:** mover el stack completo de Railway + Clerk + Postgres → Firebase App Hosting + Firebase Auth + Firestore + Firebase Storage, manteniendo el juego funcional. Reemplazar `pgvector` por Firestore Vector Search y `tsvector` por Algolia (extensión oficial Firebase).
**Acceptance:** sign-in con Google funciona en URL de Firebase App Hosting, una campaña se crea, un turno se juega de extremo a extremo, la memoria escribe y lee desde Firestore, los tests verdes, audit del subagente sin findings críticos.

---

## 1. Por qué este milestone existe

El usuario quiere lanzar el juego en producción sobre el ecosistema Firebase completo: hosting, auth, base de datos y storage bajo el mismo proveedor. La opción híbrida (Firebase + Postgres externo) y la opción Cloud SQL fueron descartadas a favor de Firestore puro, asumiendo el coste de reescribir el data layer relacional → documental a cambio de stack unificado.

**Trade-off aceptado:** mayor inversión upfront en reescritura para quedar con un único proveedor y sin dependencias de Postgres.

---

## 2. Scope

### In scope

- Firebase App Hosting reemplaza Railway como destino de despliegue.
- Firebase Auth (Google OAuth + Email/Password) reemplaza Clerk.
- Firestore reemplaza Postgres + Drizzle.
- Firestore Vector Search reemplaza `pgvector` para los embeddings (campos `embedding` actualmente null en M1; se reescribe la columna pero no hay backfill de datos).
- Algolia (Firebase extension oficial) reemplaza `tsvector` para búsqueda full-text en `recall_scene`.
- Firebase Storage queda configurado y listo (no se usa hoy; se deja preparado).
- Eliminación de dependencias muertas: `@clerk/nextjs`, `svix`, `drizzle-orm`, `drizzle-kit`, `pg`, `@types/pg`.
- Eliminación de archivos muertos: `Dockerfile`, `railway.json`, `drizzle.config.ts`, carpeta `drizzle/`, webhook `src/app/api/webhooks/clerk/`.
- Migración de secrets a Firebase Secret Manager (Anthropic, Google, OpenAI, Langfuse, PostHog, Algolia keys).
- Tests Vitest adaptados (mocks de Drizzle → mocks de Firestore Admin SDK).

### Out of scope

- Backfill de embeddings en datos existentes (no hay; las columnas estaban null en M1; el embedder es M4).
- Migración de datos de prod (no hay datos de prod; el juego no estaba lanzado).
- Cambios en agentes, prompts, o lógica narrativa. Los módulos de `src/lib/agents/` no cambian más allá de adaptar imports al nuevo data layer.
- Observability: Langfuse y PostHog siguen igual (son externos, no afectados).
- Mastra: sigue igual (compute layer, no toca DB).
- LLM providers: siguen igual (Anthropic / Google / OpenAI SDKs).

## 3. Prerequisites

Antes de la fase 1, el usuario debe tener:

1. **Proyecto Firebase creado** con Firestore, Auth (Google + Email/Password), Storage y App Hosting habilitados.
2. **Plan Blaze activado** (App Hosting y Vector Search lo requieren). Tier gratis cubre desarrollo y testing.
3. **Service account JSON** descargado y guardado en una ruta fuera del repo (p. ej. `~/.firebase/aidm-v4-service-account.json`).
4. **Web app config** (`firebaseConfig` con `apiKey`, `authDomain`, `projectId`, etc.) — son valores `NEXT_PUBLIC_*`, públicos por diseño.
5. **firebase-tools instalado** globalmente y `firebase login` ejecutado.
6. **Project ID** de Firebase para que Claude lo sepa.

### Fase 3 — Data model: schema relacional → Firestore (commits 4–6)

**Mapeo schema → Firestore.** 19 tablas en `src/lib/state/schema.ts`:

| Tabla Postgres | Colección Firestore | Notas |
|---|---|---|
| `users` | `users/{uid}` | uid = Firebase Auth uid (no se autogenera) |
| `userRateCounters` | `users/{uid}/rateCounters/{periodKey}` | subcolección |
| `userCostLedger` | `users/{uid}/costLedger/{ledgerId}` | subcolección |
| `profiles` | `profiles/{profileId}` | top-level; `ownerUid` para queries |
| `campaigns` | `campaigns/{campaignId}` | top-level; `ownerUid` para queries |
| `characters` | `campaigns/{campaignId}/characters/{characterId}` | subcolección |
| `turns` | `campaigns/{campaignId}/turns/{turnId}` | subcolección; ordenadas por `turnNumber` |
| `contextBlocks` | `campaigns/{campaignId}/contextBlocks/{blockId}` | subcolección |
| `ruleLibraryChunks` | `ruleLibraryChunks/{chunkId}` | top-level (compartido entre campañas); con campo `embedding: Vector` |
| `npcs` | `campaigns/{campaignId}/npcs/{npcId}` | subcolección |
| `locations` | `campaigns/{campaignId}/locations/{locationId}` | subcolección |
| `factions` | `campaigns/{campaignId}/factions/{factionId}` | subcolección |
| `relationshipEvents` | `campaigns/{campaignId}/relationshipEvents/{eventId}` | subcolección |
| `semanticMemories` | `campaigns/{campaignId}/semanticMemories/{memId}` | subcolección; con campo `embedding: Vector` |
| `foreshadowingSeeds` | `campaigns/{campaignId}/foreshadowingSeeds/{seedId}` | subcolección |
| `voicePatterns` | `campaigns/{campaignId}/voicePatterns/{patternId}` | subcolección |
| `directorNotes` | `campaigns/{campaignId}/directorNotes/{noteId}` | subcolección |
| `spotlightDebt` | `campaigns/{campaignId}/spotlightDebt/{debtId}` | subcolección |
| `arcPlanHistory` | `campaigns/{campaignId}/arcPlanHistory/{historyId}` | subcolección |

**Decisiones de modelado:**

- **Subcolecciones por campaña** para todo lo "scoped a una campaña". Pros: queries naturales (`campaigns/{id}/turns`), borrado en cascada vía script, security rules limpias. Cons: collection group queries cuando hace falta listar across campañas (Firestore las soporta, no es bloqueante).
- **Zod en cada I/O.** Firestore no enforza shape; el contrato vive en `src/lib/types/`. Helpers tipados en `src/lib/firebase/firestore.ts` parsean al leer.
- **Sin foreign keys.** Las relaciones se modelan por campos `*Uid` o IDs. Integridad referencial es responsabilidad de la app.
- **Embeddings:** campos `embedding` se declaran como `FieldValue.vector(...)` cuando se llenen (M4). Los índices vectoriales se crean en commit 6.
- **Timestamps:** `serverTimestamp()` para `createdAt`/`updatedAt`. Drizzle usaba `defaultNow()`.
- **Counters / aggregations:** Firestore no tiene `COUNT(*)` barato. Para `userRateCounters` y `spotlightDebt` se usan documentos contador con `FieldValue.increment()`.

## 5. Riesgos

- **Firestore Vector Search es relativamente nuevo.** Los índices vectoriales tardan en construirse y la API puede cambiar. Mitigación: como los embeddings están null en M1 y M4 está lejos, podemos diferir la creación de índices hasta que haya datos.
- **App Hosting + Next 15 middleware.** Firebase tiene buen soporte de Next 15 pero hay edge cases con middleware y dynamic routes. Mitigación: probar en dev antes del primer commit de producción; si algo no funciona, fallback a Cloud Run directo (mismo runtime).
- **Algolia tier free** tiene límites (10k records, 10k operations/mo). Suficiente para testing; si la campaña crece mucho, hay que pagar o tokenizar manualmente.
- **Firestore costs.** Lecturas se pagan por documento. Patrones que en SQL son un JOIN barato pueden ser N lecturas. Mitigación: denormalización donde haga sentido, vigilar billing.
- **Tests de integración.** Vitest contra Postgres real fue la práctica; ahora hay que usar el emulador de Firestore. Setup adicional pero la práctica del usuario (CLAUDE.md "no mock the database") sigue válida — el emulador es una BBDD real.
