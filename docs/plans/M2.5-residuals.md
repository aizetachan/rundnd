### 2.1 Campaign archive cron

**Files:**
- `scripts/campaigns-archive-cron.ts` — new script, mirrors `scripts/users-hard-delete.ts` shape:
  - Scans `campaigns.where(deletedAt < now - 14d)`.
  - For each match: purge every subcollection (CAMPAIGN_SUB.* — 14 of them) via batched 250-doc deletes, then delete the campaign doc itself.
  - `--days <n>` flag for testing (default 14).
  - `--dry-run` prints the plan without deleting.
  - `--confirm` required for actual delete (refuses without it unless `--dry-run`).
  - Best-effort Algolia cleanup mirrors the users script (no-op today, hook reserved for future delete-by-filter).
- `package.json` — add `"campaigns:archive": "tsx --env-file=.env.local scripts/campaigns-archive-cron.ts"`.

**Coordination with `users-hard-delete`:** the two scripts operate on orthogonal axes (users vs campaigns), but a soft-deleted user who also had soft-deleted campaigns could in theory have those campaigns purged by either script depending on cron timing. The end state is identical ("deleted"), so dedup isn't required — both are idempotent.

**Cloud Scheduler still deferred.** Per the M3 plan §3 and the M2.5 retro: Cloud Scheduler integration lands at M3.5+ (admin surface) or M9 (auditable retention for billing). Today, manual run.

## 4. Risks

- **First-run blast radius.** A long-soft-deleted user testing the archive cron with `--days 0` could nuke all their campaigns. Mitigation: `--confirm` is required; default `--days 14`; `--dry-run` always available; pattern is borrowed verbatim from `users-hard-delete`.
- **Docstring rewrites cause merge friction.** Both files are touched frequently in agent work; if the diff lands during another in-flight edit, conflicts are mechanical. Single-commit timing makes that low-risk.
