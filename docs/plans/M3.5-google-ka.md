**Milestone:** M3.5 (after M3 close + Wave B sub 6+7).
**Status:** 🟡 Plan written, sub 1 implementation in flight.
**Authority:** ROADMAP §23 → "M3.5 Google-KA"; `src/lib/agents/key-animator.ts:222` provider guard that documents "Google-KA lands M3.5"; `src/lib/providers/google.ts` (provider slot stubbed `available: false` with `unavailableReason: "Google-KA (Gemini-native) lands at M3.5"`).
**Goal:** campaigns configured with `provider="google"` can play a turn end-to-end on Gemini 3.1 Pro. The conductor / handoff / memory layers stay provider-agnostic; only KA's substrate changes.

---

## 1. The shape of KA on Anthropic vs Google

KA on Anthropic runs on the **Claude Agent SDK**, which gives:
- A streaming `query()` loop that handles tool calls automatically.
- Native MCP servers passed via `options.mcpServers` (the 7 cognitive memory layers + entity tools).
- Custom subagents ("consultants") via `options.agents` — KA invokes OutcomeJudge / Validator / Pacing / Combat / MemoryRanker / Recap / ScaleSelector through the Agent tool.
- A `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` marker that splits cached blocks 1–3 from per-turn block 4.

Gemini, via `@google/genai`, gives:
- A streaming `models.generateContentStream()` loop.
- **No native MCP.** Tools are exposed as Function Declarations; the loop manages tool-call rounds manually.
- **No subagent primitive.** Consultants land as manually-orchestrated function calls.
- A `cachedContent` API for prompt caching (context-id model, different from Anthropic's breakpoint model).
- Native `thinkingConfig` on Gemini 3 Pro.

**This is a substrate change, not a config flip.** A full Google-KA needs a re-implemented orchestration loop. Doing it correctly takes multiple sub-commits.

## 2. Sub-commits

### Sub 1 — Foundation + minimal streaming Google-KA (this commit)

Ships the substrate, dispatch, and a narration-only path. Players on a Google campaign can complete a turn — KA streams a Gemini response based on the rendered blocks — but without tool calls or consultants. Memory writes, OutcomeJudge verdicts, Validator checks, etc. don't fire for Google campaigns at sub 1; they ride on the post-turn workers as today (those don't depend on KA's substrate) for any pieces that fit, and the rest is documented gap.

**Files:**
- `src/lib/agents/ka/index.ts` — dispatcher. Re-exports `runKeyAnimator`, `KeyAnimatorInput`, etc. Picks `runKeyAnimatorAnthropic` or `runKeyAnimatorGoogle` based on `input.modelContext.provider`.
- `src/lib/agents/ka/anthropic.ts` — the existing implementation, lifted from `key-animator.ts`. Renamed exports `runKeyAnimatorAnthropic`. The guard against non-anthropic providers is removed from this file (the dispatcher now routes).
- `src/lib/agents/ka/google.ts` — NEW. Gemini 3.1 Pro via `@google/genai`. Renders the same blocks 1–4 system prompt (joined with a string boundary marker instead of Anthropic's enum constant), streams text deltas, yields `KeyAnimatorEvent`s with the same shape Anthropic-KA does. NO tools / no functionDeclarations at sub 1. Cost reported from Gemini's response.usageMetadata.
- `src/lib/agents/key-animator.ts` — slimmed to a re-export of the dispatcher. Preserves the existing import surface (`runKeyAnimator`, `KeyAnimatorInput`, etc.) so callers don't change.
- `src/lib/providers/google.ts` — flip `available: true`. Update `unavailableReason` → remove, since available.
- Tests: `src/lib/agents/ka/__tests__/google.test.ts` — minimal: streams text, yields final, errors when API key missing. `src/lib/agents/ka/__tests__/dispatch.test.ts` — routes anthropic vs google based on provider, throws for openai/openrouter.

**Out of scope at sub 1** (documented as sub 2):
- Tool calls (entity registration, memory writes from inside KA).
- Consultant subagents (OutcomeJudge / Validator / etc. spawning from inside the Gemini loop).
- Prompt caching via Gemini's `cachedContent` API.
- Per-block fingerprint recording (still records as one combined block).

### Sub 2 — Tool calls + consultants on Google-KA (future commit, NOT this plan)

When sub 1 ships and we have telemetry from real Gemini turns, sub 2 ports the MCP tools as Function Declarations and orchestrates consultant calls manually inside the loop. Could be a multi-day effort; gets its own plan.

## 3. Scope decisions (sub 1)

- **Minimal narration-only Google-KA at sub 1.** Per CLAUDE.md "don't defer load-bearing structure to later milestones" — but the load-bearing structure here is the dispatcher + substrate split, which DO land at sub 1. Tools and consultants are content, not structure. Sub 2 fills them in.
- **Keep `src/lib/agents/key-animator.ts` as a re-export shim.** All existing imports (`@/lib/agents/key-animator`) still work. Diff stays small.
- **No prompt caching at sub 1.** Gemini's `cachedContent` is a context-id model (create once, reference by id); needs a TTL cache that the per-turn flow doesn't have today. Defer to sub 2.
- **Provider guard removed for "google" only.** OpenAI / OpenRouter still throw — those land M5.5.
- **Don't change conductor / handoff / memory layers.** The conductor at /sz runs on Sonnet 4.6 regardless of campaign provider. Memory writes happen post-turn from worker queue (M5+); KA's substrate doesn't gate those.

---

## 4. Risks

- **Voice quality on Gemini may differ.** Players on Google campaigns get a different author. That's the point — we're enabling provider choice. The eval harness from Wave B sub 7 could be extended to score narration quality cross-provider once judge axes ship (deferred).
- **Cost reporting from Gemini's `usageMetadata` is approximate vs Anthropic's `total_cost_usd`.** We compute cost manually from token counts + pricing snapshot. Slight drift vs Langfuse's billed cost.
- **Streaming shape differences.** Gemini's `generateContentStream` returns full chunks; Anthropic returns deltas. Both adapt to the same `{kind: "text", delta}` shape but chunk boundaries differ. Player experience: visually identical (text flows in); telemetry: TTFT is to first-non-empty-chunk on both.
- **No tool calls means no Chronicler writes.** A Google-campaign turn at sub 1 won't write to memory layers (`write_semantic_memory`, `register_npc`, etc.) from inside KA. Post-turn workers (memory writer, Chronicler) still fire on the rendered narrative — coverage gap is small.
