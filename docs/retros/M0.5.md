## 2. Decisiones de diseño que vale la pena anotar

### 2.1 Auth

- Cookie de sesión httpOnly (`__session`, 14 días) creada con `auth.createSessionCookie` del Admin SDK; cliente solo maneja ID tokens efímeros.
- Middleware en Edge runtime hace **presence check** de la cookie, no verificación criptográfica (Admin SDK necesita Node). Verify real ocurre en route handlers / Server Components vía `getCurrentUser`.
- `auth_time < 5 min` enforces freshness al mintear la cookie (anti-replay).
- `detail` field en errores 400/401 gateado por `NODE_ENV !== "production"` — no oracle de validez de tokens en prod.

### 2.2 Modelo de datos

- **Top-level `campaigns`** (no subcolección de `users`) — permite collection-group queries y rules más limpias por `ownerUid`.
- **Subcolecciones bajo `campaigns/{id}`** para todo lo scoped a campaña: turns, npcs, locations, contextBlocks, semanticMemories, etc.
- **Slug como doc id** para profiles + ruleLibraryChunks + register_npc/location/faction. Race-free idempotencia.
- **Doc id = npcId** en spotlightDebt para enforce (campaign, npc) uniqueness estructuralmente.
- **Zod en cada I/O.** Firestore no enforza schema; el contrato vive en `src/lib/firestore/schemas/`.
- **Embeddings:** campos `embedding` declarados pero null. M4 backfillea. Vector Search se configura cuando haya datos.

### 2.3 Concurrency

- **Locks via flag-doc + timeout** (15s para turns, 60s para chronicler) en lugar de `pg_advisory_lock`. Comportamiento: back-off (`skipped_concurrent`) en lugar de queueing. Suficiente para single-player games.
- **Atomic counters** con `FieldValue.increment` + `set merge`. Rate limiter + cost ledger funcionan igual que con Postgres ON CONFLICT.
- **Transactions** para auth-create-merge en `ensureUserSeeded` (preserva `dailyCostCapUsd` cuando ya existe — `set merge` con null lo clobberearía).

### 2.4 Recall_scene degradado

`tsvector` no tiene equivalente nativo en Firestore. Implementación temporal: pull de los últimos 50 turns + substring match case-insensitive en JS, score binario. Fase 4 (Algolia extension oficial) lo sustituye.

### 2.5 Hosting

- App Hosting nativo de Next.js 15 — no `output: standalone` ni `Dockerfile`.
- Secrets via Secret Manager (`firebase apphosting:secrets:set`); env vars públicos como literales en `apphosting.yaml`.
- Reglas Firestore default-deny; lecturas client-side solo para el propio user + sus campañas + corpora públicos (profiles, ruleLibraryChunks). Admin SDK bypasses las rules para todas las escrituras.

## 4. Pasos manuales pendientes para el usuario

1. **Configurar secrets en Secret Manager** (uno por LLM key + observability):
   ```bash
   firebase apphosting:secrets:set ANTHROPIC_API_KEY
   firebase apphosting:secrets:set GOOGLE_API_KEY
   firebase apphosting:secrets:set OPENAI_API_KEY
   firebase apphosting:secrets:set OPENROUTER_API_KEY
   firebase apphosting:secrets:set LANGFUSE_PUBLIC_KEY
   firebase apphosting:secrets:set LANGFUSE_SECRET_KEY
   firebase apphosting:secrets:set NEXT_PUBLIC_POSTHOG_KEY
   # Algolia keys quedan para Fase 4
   ```

2. **Conectar GitHub repo a App Hosting backend.** En la consola Firebase:
   - Build → App Hosting → Get started
   - Backend ID: `rundnd` (o el que prefiera)
   - Region: pick la que coincida con la región de Firestore
   - Repo: `aizetachan/rundnd`, branch `main`
   - Root: `/`
   - Live branch: `main`

3. **Deploy reglas Firestore + Storage:**
   ```bash
   firebase deploy --only firestore:rules,storage:rules,firestore:indexes
   ```

4. **Smoke test post-deploy:**
   - Abre la URL de App Hosting (algo como `https://rundnd--rundnd-fd1dc.<region>.hosted.app`)
   - Sign-in con Google
   - Verifica que `users/{uid}` se cree en la consola Firestore
   - Verifica que `campaigns/{newId}` con la Bebop seed exista
   - Click en la campaign → verifica que carga (puede pedir un índice composite la primera vez; click en el link del error y re-deploy)
   - Intenta jugar un turno (necesita ANTHROPIC_API_KEY funded)

## 5. Riesgos conocidos

- **Coste Firestore.** Lecturas se pagan por documento. Patrones que en SQL eran un JOIN barato son N lecturas. El recall degradado de scene lee 50 turns por consulta — vigilar billing si los logs muestran muchas llamadas. Algolia (Fase 4) lo arregla.
- **Concurrency model distinto.** Postgres `pg_advisory_lock` encolaba turns; Firestore back-off los rechaza con `skipped_concurrent`. Para single-player es fine; revisitar si hay multiplayer en el futuro.
- **`heat` puede exceder 100** brevemente. Postgres clampaba con `LEAST(100, …)`; Firestore `FieldValue.increment` no tiene primitiva de ceiling. La siguiente pasada de decay lo regresa al rango. Floor invariant (`heat >= 0`) sigue.
- **Eval harness desactivado.** `pnpm evals` sale clean con TODO. Si el usuario depende de evals para PR gates, es bloqueante.

## 6. Lecciones para el próximo milestone

- **Empezar con un repo en git** ahorró bastante stress más adelante. La primera sesión no tenía git inicializado y eso era una bomba de tiempo.
- **Dual-mode (`db` + `firestore` ambos en `AidmToolContext`)** durante una migración de varios sub-commits funcionó muy bien. Permitió migrar tools uno a uno con tests verdes en cada paso. Cuando el último sub-commit eliminó `db`, el cleanup fue trivial.
- **Slug-as-doc-id** es el truco más útil para preservar uniqueness sin transactions. Race-free, idempotente, lookup directo. Aplicado en profiles, npcs, locations, factions.
- **Audit del subagente** capturó dos bugs de seguridad reales en Fase 2 (open-redirect, error-detail leak) que valen el costo del audit. Fase 3 sub-commits 4-6 no se auditaron por velocidad — vale la pena un audit retroactivo antes del primer deploy productivo.
