## 2. Architecture decisions worth keeping

### 2.1 Export schema is `v1` and bundle-shaped

The bundle has `schema_version: "v1"` at the root so a future import path can branch on the version. The shape is one big nested object — `user → campaigns[] → turns[] / characters[] / context_blocks[] / semantic_memories[] / session_zero` — rather than parallel arrays keyed by id. Nested is uglier on disk but trivial to traverse + tells a coherent story to a human reader. Re-import (M3+) can normalize back into the Firestore subcollection shape.

What stayed out of the bundle, intentionally:
- Provisional memory writes (the writer doesn't exist yet at M3; lands M3+).
- OpeningStatePackage versioned artifacts (verbose; recoverable from `session_zero` + the latest turn).
- Algolia index records (recoverable from Firestore sources).
- Other users' data — there's no admin export.

### 2.2 Soft delete reads outside the transaction

`softDeleteUser` queries the owned campaigns BEFORE opening the transaction so the transaction's read set stays small. Race window — a new campaign created between the query and the commit — survives the soft-delete pass; the hard-delete cron mops it up later by scanning `where(ownerUid==X)` regardless of `deletedAt`. The trade is "rare orphan that gets cleaned up in 24h" vs "transaction limited to ~500 ops total." Right trade.

### 2.3 getCurrentUser checks deletedAt at runtime

Firebase Auth session cookies are 14-day TTL. A soft-deleted user's cookie stays valid until the cookie itself expires. Without a runtime gate, that user could still hit auth-required APIs for ~14 days post-delete.

Two options were on the table:
- A: Revoke the Firebase Auth session at delete time. Firebase Auth supports per-user revocation via Admin SDK's `revokeRefreshTokens` — but session cookies are server-side credentials separate from refresh tokens, and revocation semantics are uneven across SDK versions.
- B: Add a Firestore read in `getCurrentUser` that checks `users/{uid}.deletedAt`.

Took B. One extra Firestore read per request is comparable latency to the `verifySessionCookie(checkRevoked=true)` we already pay; Firestore reads are cheap; the seam is explicit and testable.

Fail-open on Firestore unavailability: if the read throws (rare), we let the request proceed. The campaigns queries downstream filter by `deletedAt==null` anyway, so a stale request from a deleted user sees "looks empty," not "data leak."

### 2.4 Hard-delete is a manual cron script

Per the plan §3: Cloud Scheduler integration deferred. The script lives at `scripts/users-hard-delete.ts` with `--dry-run` and `--confirm` flags so a misclick doesn't nuke data. M9 (Stripe billing) or M3.5+ (admin surface) will likely promote this to a scheduled job; today it's manual.

The script iterates 12 campaign subcollections + 2 user subcollections (rateCounters + costLedger) per user. 250-doc batched deletes stay under Firestore's 500-op transaction cap. Idempotent — re-running on the same user is a no-op.

## 3. What surprised us

### 3.1 The export bundle is straightforwardly portable

A 10-turn campaign exports as ~50–80 KB JSON. A hypothetical 200-turn campaign would be 500 KB – 1 MB. Streaming response handles it cleanly; the client-side blob download works in Safari/Firefox/Chrome. No back-pressure issues observed.

### 3.2 Firebase Auth doesn't have a clean per-user session-cookie revoke

The `verifySessionCookie(checkRevoked=true)` only checks against refresh-token revocation; session cookies that were minted before the revocation can still verify. The `deletedAt` Firestore gate sidesteps that — but it means a deleted user's cookie still passes verifySessionCookie until natural expiration. The runtime gate is what closes the door.

### 3.3 12 campaign subcollections is a lot to keep in sync

The hard-delete script iterates `Object.values(CAMPAIGN_SUB)` so a future-added subcollection automatically gets purged. Same pattern works for user subcollections via `USER_SUB`. This is one of those "the constants file finally pays off" moments — adding a new subcollection costs 0 lines in the deleter.

## 6. Where the codebase stands now

The user lifecycle is closed end-to-end:

```
sign-up via Google
  ↓
land on /campaigns (empty for new accounts)
  ↓
Session Zero for any IP (Wave A + B)
  ↓
play turns (KA streaming, Chronicler post-turn, recall_scene via Algolia)
  ↓
inspect memory (/campaigns/[id]/memory)
  ↓
adjust spending cap, export data, OR delete account (/account/spending)
  ↓
hard-delete cron runs 24h later
```

What's missing for "complete game experience" past this point:
- Image generation (M5+).
- Custom profile authoring UI (separate plan).
- Path A research scrapers + eval harness (Wave B sub 6+7 coupled).
- Google-KA / OpenAI-KA (M3.5 / M5.5).
- Embedder + Vector Search (M4).

None of those are gaps in the playable text-only loop — they're new features / new providers / new modalities.

## 7. Tone

Two sub-commits, no surprises. The pattern from M2.5 sub 3+4 (read-only UI work) extended naturally into M3 sub 1 (export). The harder design was sub 2 — picking the deletedAt gate over Firebase Auth revocation — and that one decision is documented in §2.3 for the next person who wonders why we didn't just call `revokeRefreshTokens`.

Three milestones (M2 Wave A, M2 Wave B, M2.5, M3) in close succession. The codebase has its hardening + polish + lifecycle layers in place. Next plan can be a real feature push — M4 embedder, M5+ image gen, or M3.5 Google-KA — without "but first we need to fix X" debt blocking it.
