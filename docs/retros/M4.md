## 2. Architecture decisions worth keeping

### 2.1 Embed at write, not read

Every `write_semantic_memory` call blocks on the embedder (~50–100ms). Read path (`search_memory`) pays one embed per query (or per fan-out leg) and then does a single `findNearest`. Reads stay cheap; writes amortize the embedder cost into the natural Chronicler latency.

Trade-off: if the embedder is down, writes still succeed (try/catch persists `embedding: null`). Backfill catches them later.

### 2.2 `findNearest` candidate count `k * 2`, code-side filter to `k`

The vector query at the Firestore layer pulls `min(k * 2, 40)` candidates. We then filter by category + min_heat + apply STATIC_BOOST + sort by computed relevance and truncate to `k`. Putting the secondary filters in code avoids spawning a composite vector + range index per category in `firestore.indexes.json`, which would be a 16-index combinatorial mess.

### 2.3 Cosine distance → relevance via `1 - distance / 2`

Cosine distance is in [0, 2] (0 = identical, 2 = opposite). Mapping to a [0, 1] relevance score lets STATIC_BOOST add cleanly. Heat factor multiplies (0.5 + heat/200) so 100-heat memories preserve their raw similarity while 0-heat memories see a 50% relevance haircut.

### 2.4 Heat boost on access is fire-and-forget

`boostHeatOnAccess` writes for every returned memory but the read path doesn't await them. A failed boost is a missed +N heat point (recovers on the next access); preventing it from blocking the turn was worth more than the write durability.

### 2.5 Backfill is a collection-group walk, not a query

Scanning `collectionGroup("semanticMemories")` and filtering `embedding == null` in code sidesteps the single-field-index requirement for the null query. Works at AIDM scale; at 100k+ docs we'd promote to an indexed query.

### 2.6 Memory page shows embedded vs not

The inspection page (`/campaigns/[id]/memory`) renders a small chip per semantic memory: `embedded` (green) or `no embed` (muted). User can see at a glance whether their campaign's memory is participating in vector recall or sitting on category-only fallback. Pushes backfill-needed signal up to the surface where the user can act on it.

## 3. What surprised us

### 3.1 Firestore vector index format

The `firestore.indexes.json` shape for a vector index is `{ "fieldPath": "embedding", "vectorConfig": { "dimension": 768, "flat": {} } }`. Field-overrides aren't required; the index goes in the regular `indexes[]` array under `collectionGroup: "semanticMemories"`. The `flat: {}` is the index variant — Firestore today only supports flat (vs hierarchical) vector indexes.

### 3.2 Gemini embed responses don't carry token counts

Other LLM SDKs (Anthropic, OpenAI) return `usage.input_tokens`. Gemini's `EmbedContentResponse` does not. We approximate via char-length / 4 for the cost ledger entry. Acceptable upper-bound; if Google adds the field later we swap to the real count.

### 3.3 `findNearest` exists on Admin SDK but `VectorQuery` type isn't re-exported

`firebase-admin/firestore` doesn't re-export the `VectorQuery` type from `@google-cloud/firestore`. We dropped the explicit type annotation and let TS infer from the chained call. Works fine; would be cleaner if Admin re-exported the type.

## 6. Where the codebase stands now

A new player writing semantic memory ends up with:
1. `write_semantic_memory({ category, content, ... })` fires.
2. `embedText(content)` returns a 768-dim vector.
3. The Firestore doc lands with `embedding: [...]`.
4. KA's next `search_memory("Vicious motives", k=5)` embeds the query, runs `findNearest`, applies boost + filter, returns ranked results.

The seven cognitive layers are no longer a stub. Storyboarded fragments + vector recall are the working substrate for the rest of the long-tail features (M5 image generation, M5.5 OpenAI-KA, M6 multi-character ensembles).

## 7. Tone

Sub 1 was the hardest commit because it threaded the embedder into a code surface that had been speaking `embedding: null` for months without flinching. Sub 2 made the read path useful by actually consuming what sub 1 wrote. Sub 3 caught the rows born before sub 1. Sub 4 added the small UI affordance that turns "is it embedded?" from a database question into a surface anyone can read.

The hard work was the architectural choice to embed-at-write (not read). Everything downstream follows from that decision being correct.
